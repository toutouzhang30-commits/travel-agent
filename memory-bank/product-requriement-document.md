# 旅游 AI 助手 PRD（一页纸初稿）

## 1. 产品名称
旅游 AI 助手（MVP）

## 2. 产品定位
一个面向自由行用户的 Web 聊天式旅游规划助手。
核心价值不是泛泛回答旅游问题，而是帮助用户把已确定目的地快速细化成少修改可用的可执行行程。

## 3. 目标用户
- 自由行用户
- 已经基本确定目的地，但不想自己花大量时间查攻略、拼路线、做预算的人群

## 4. 核心场景
用户进入产品后，直接在聊天框输入需求，例如：
- “我下个月去杭州玩 3 天，想轻松一点，预算 3000”
- “北京 5 天，想看人文景点和吃本地美食”
- “上海周末两天，偏拍照打卡”

AI 根据用户输入完整度动态追问，逐步补齐信息，并生成按天行程卡。
在后续阶段，系统还应能通过知识检索和工具调用补充更可靠的信息。

## 5. 核心价值主张
### 对用户
- 省时间：减少查攻略、比路线、做预算的时间成本
- 更可执行：输出的是按天可落地的行程，不是泛泛建议
- 更可靠：显示来源，关键数据不确定时明确提示
- 更易迭代：用户可继续通过聊天修改方案

### 对产品
- 先聚焦高频旅游规划场景，验证“AI 行程细化”是否有真实需求和使用价值
- 通过网页聊天室 + 流式过程展示，验证用户是否真的需要“边生成边看过程”的体验

## 6. MVP 范围
### 首发渠道
- Web 聊天页

### 首发城市试点
- 北京
- 上海
- 杭州

### 选择原因
- 用户需求多
- 旅游场景丰富，适合验证城市知识检索与行程生成能力

### 支持范围
- 行程天数：不限制
- 预算风格：
  - 经济型
  - 均衡型
  - 品质型
- 旅行风格：支持多种风格偏好输入和生成

## 7. 核心功能
### 7.1 聊天式需求收集
- 用户直接进入聊天框输入需求
- AI 根据输入完整度动态追问
- 重点理解：
  - 预算风格
  - 兴趣偏好
  - 日期/天数
  - 节奏偏好
  - 同行情况（可扩展）

### 7.2 行程生成
生成按天行程卡，可包含：
- 日期 / 第几天
- 上午 / 下午 / 晚上安排
- 景点名称
- 推荐理由
- 交通方式与耗时
- 每日预算估算
- 注意事项
- 备选方案

### 7.3 行程修改
- 用户通过继续聊天的方式修改行程
- 例如：
  - “第二天太累了，轻松一点”
  - “不要博物馆，多安排吃的”
  - “预算再压低一点”
  - “如果下雨给我替代方案”

### 7.4 预算估算
- 按天估算预算
- 支持不同预算档位下的行程建议差异化

### 7.5 替代方案
- 关键场景下给出备选安排
- 例如：
  - 景点拥挤
  - 下雨
  - 预算不足
  - 体力安排不合理

### 7.6 来源展示
- 对景点、攻略、营业时间等尽量显示来源
- 提高用户对结果的信任感
- 对缺乏可靠来源的信息，明确提示不确定性
- 对网页采集来源，需要展示来源名称、链接、采集/校验时间和可信度提示

### 7.7 流式过程展示
- 在网页聊天页中逐步展示用户可见的执行过程
- 至少包括：
  - 正在分析需求
  - 正在补齐需求
  - 正在生成行程
  - 后续可扩展为检索状态和工具调用状态

## 8. 数据与质量要求
### 最关键的数据来源
- 景点 / 攻略信息
- 经过治理的公开网页信息
- 后续阶段补充天气、路线、价格等高时效数据

### 首版质量原则
当关键数据不可靠时：
- 必须明确提示不确定性
- 不应假装准确
- 不应把未经清洗和校验的网页内容直接当作可靠结论
- 不应把实时天气、票价、路线耗时写入静态 RAG 知识库

### 最不能接受的错误
- 景点没开门还推荐
- 路线明显绕路
- 预算严重失真
- 把缺乏来源的信息伪装成确定事实
- 用户明确要求修改后，最终行程没有真正体现修改

### 质量闭环
- 系统应具备最小 Reflection 能力，对生成或修改后的行程做约束校验
- 当发现预算、节奏、天数、来源或工具结果存在明显问题时，应提示不确定性或进行局部修正

## 9. 明确不做
首发 MVP 暂不包含：
- 直接预订
- 多人协作
- 实时在途重规划
- 复杂账户体系
- PDF 导出
- 复杂前端框架重构

## 10. 成功标准
### 产品成功标准
- 用户拿到的行程能够做到少修改可用
- 用户可以在网页端顺畅完成“输入需求 → 追问补齐 → 生成行程 → 继续修改”的闭环
- 用户能看到关键建议背后的来源，并理解哪些信息需要二次确认

### 核心观察指标
- 用户继续追问修改的次数
- 流式聊天主链路是否稳定可用
- RAG 检索命中质量与来源可用率
- Reflection 发现并修正明显问题的比例

含义：
- 如果生成结果越接近用户预期，后续修改轮次应越少
- 如果流式聊天过程足够清晰，用户更容易理解系统当前在做什么

## 11. 一句话总结
这是一个面向自由行用户的 Web 旅游 AI 助手，聚焦北京、上海、杭州试点城市，帮助用户通过聊天快速把已确定目的地细化成可靠、可执行、可迭代的按天行程方案，并逐步演进到具备来源展示、知识检索与工具调用能力的旅游代理。
