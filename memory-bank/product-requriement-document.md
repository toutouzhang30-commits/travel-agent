# 旅游 Agent PRD（统一阶段版）

## 1. 产品名称
旅游 Agent（MVP）

## 2. 产品定位
这是一个面向自由行用户的 Web 旅游规划助手，但它的目标已经不再是“泛化聊天 + 顺手生成一份行程”，而是一个能够：

- 理解用户真实意图
- 主动补齐缺失需求
- 检索相对可信的旅游知识
- 调用实时工具获取时效信息
- 对结果做最小反思和修正
- 用聊天方式持续迭代 itinerary

它要解决的核心问题不是“回答旅游问题”，而是帮助用户把**已经大致确定的目的地**快速细化成**更可靠、可执行、可修改**的行程方案。

---

## 3. 目标用户
### 3.1 核心用户
- 自由行用户
- 已基本确定要去哪里，但不想自己长时间查攻略、比路线、做节奏安排的人
- 希望一边对话一边把行程逐步收敛，而不是一次性拿到一份无法继续修改的结果的人

### 3.2 当前优先支持的使用者画像
- 第一次去某城市，希望先得到节奏建议和区域建议
- 已经知道大概预算和天数，希望快速生成初版 itinerary
- 已经有初版 itinerary，希望继续修改得更轻松、更省钱、更符合兴趣偏好
- 对天气、路线、票价等信息有进一步确认需求的人

---

## 4. 核心场景
### 4.1 首轮需求收集
用户输入：
- “我下个月去杭州玩 3 天，想轻松一点，预算 3000”
- “北京 5 天，想看人文景点和吃本地美食”
- “上海周末两天，偏拍照打卡”

系统应：
- 提取当前有效需求
- 判断还缺什么关键信息
- 继续追问，而不是过早生成泛泛方案

### 4.2 旅游知识问答
用户输入：
- “上海拍照打卡适合住哪里？”
- “北京第一次去怎么安排更顺？”
- “杭州雨天怎么玩？”

系统应：
- 优先走知识检索，而不是靠关键词硬分流
- 在回答中尽量带来源和边界说明

### 4.3 实时工具问答
用户输入：
- “杭州明天会下雨吗？”
- “上海迪士尼门票多少钱？”
- “从西湖到灵隐寺大概要多久？”

系统应：
- 将这类问题识别为实时或半实时问题
- 优先走工具层，而不是让 RAG 或模型自由编造
- 工具不可用时明确说明不确定性

### 4.4 itinerary 生成与修改
用户输入：
- “杭州 3 天，预算 3000，轻松一点，想吃美食和看自然风景”
- “第二天太累了，轻松一点”
- “不要博物馆，多安排吃的”
- “预算再压低一点”

系统应：
- 生成按天、按时间段组织的 itinerary
- 修改时优先局部调整，而不是每次都整份重写
- 在必要时补充知识检索或工具结果

---

## 5. MVP 范围
### 5.1 首发渠道
- Web 聊天页
- 后端同步 / SSE API 继续保留，主要服务网页端和调试路径

### 5.2 首发城市试点
- 北京
- 上海
- 杭州

### 5.3 保留三城试点的原因
- 这三个城市旅游需求密集、场景丰富
- 适合验证知识检索、实时工具和 itinerary 生成的协同效果
- 能控制知识库治理和工具边界，避免一开始就把城市范围铺得过大

### 5.4 MVP 内支持的能力边界
- 聊天式需求收集
- 结构化 itinerary 生成
- itinerary 修改
- 静态/半静态旅游知识回答
- 部分实时工具能力（优先天气）
- 来源展示与不确定性提示
- SSE 流式展示用户可见的执行过程
- 最小反思-修正闭环

### 5.5 MVP 明确不做
- 直接预订
- 多人协作
- 复杂账户体系
- 实时在途重规划平台化
- PDF 导出
- 复杂前端框架重构

---

## 6. 产品核心能力
### 6.1 语义路由，而不是关键词分类
系统不应继续依赖“关键词/限定词列表”来判断问题类型。长期目标应是：
- 通过模型结构化路由识别用户当前意图
- 区分需求补齐、知识问答、工具调用、行程生成、行程修改等动作
- 解析失败时走显式 fallback，而不是偷偷按普通聊天处理

### 6.2 Working Memory
系统需要短期工作记忆来理解连续对话，而不只是保存 requirement 和 itinerary。

最少应能记住：
- 当前需求画像
- 当前 itinerary 摘要
- 最近几轮用户修改意图
- 最近一次知识检索摘要
- 最近一次工具调用结果与失败状态
- 最近一次反思结论

### 6.3 混合知识能力
知识能力不应只有“本地向量库命中几段文本”这一条路径。

目标路线应是：
- 内部知识库承载稳定旅游知识
- 外部搜索/浏览能力补足 freshness 和覆盖率
- 两路结果统一进入来源、排序和压缩流程
- 最终回答仍由编排器受控整合

### 6.4 Spring AI `@Tool` 主工具层
工具能力的当前主方向应当是 Spring AI `@Tool`，而不是继续堆很多零散的本地 Service 分支。

但文档要明确：
- 当前代码里还没有真实 MCP 接入
- MVP 当前优先推进 Spring AI `@Tool` 主工具层
- 第一阶段先通过统一工具网关把已有天气能力纳入受控工具主链路
- MCP 仅作为后续可选外部工具协议扩展
- 后续再逐步接入 maps / pricing 等更多工具；如确有需要，再评估 MCP search / browser 能力

### 6.5 反思-修正闭环
系统不应只生成结果，还应在关键节点进行最小反思：
- 结果是否满足用户核心约束
- 是否错误地把知识问题走成工具或把实时问题走成 RAG
- 是否缺少来源或把失败结果伪装成成功
- itinerary 是否真的反映了用户修改要求

当发现明显问题时，应：
- 补一次追问
- 补一次检索或工具调用
- 或对结果做一次局部修正

### 6.6 来源与不确定性
系统输出必须能回答：
- 信息从哪里来
- 是否足够可靠
- 哪些地方仍需要用户二次确认

这在旅游场景里是关键要求，尤其是：
- 实时天气
- 票价
- 路线耗时
- 开放时间
- 强依赖外部公开网页的信息

### 6.7 流式可见过程
网页端应展示用户可见、可验证的执行过程，例如：
- 正在分析需求
- 正在补齐需求
- 正在检索上海住宿区域信息
- 正在调用天气工具
- 正在校验结果
- 正在修正行程结果

不把隐藏推理过程直接暴露给用户。

## 7. 统一阶段描述：目标态 vs 当前现状
### 7.1 目标态
目标中的旅游 Agent 应具备：
- 结构化路由
- 混合 RAG
- Spring AI `@Tool` 主工具层
- MCP 后续可选扩展
- Working Memory
- Reflection loop
- 来源与不确定性闭环
- 按会话持续修改 itinerary 的能力

### 7.2 当前现状
当前仓库已经具备：
- Web 聊天页与 SSE 主链路
- 最小需求抽取与 itinerary 生成/修改
- 最小 RAG 闭环
- 来源区与检索区展示基础
- 本地天气工具调用底座
- WeatherTool 受控工具调用闭环
- 基于高德 Amap 的 MapsTool 第一版调用闭环，可回答路线、距离、交通耗时类实时问题

但仍存在明显差距：
- 结构化路由已经进入主链路，但仍需要继续验证多轮上下文下的稳定性
- Spring AI `@Tool` 主线已经覆盖 WeatherTool 和 MapsTool，PricingTool 仍未真实接入
- MCP 尚未落地，且当前不是必须落地项
- RAG 已开始进入候选治理链路，但仍需完成收口、评估和来源治理
- 短期记忆太薄
- 反思-修正闭环尚未进入主链路

因此，这个 PRD 既描述目标产品，也约束迁移方向，避免把当前骨架误当成目标形态。

## 8. 数据与质量原则
### 8.1 最关键的数据来源
- 已治理的内部旅游知识文档
- 经筛选和标注的公开网页内容
- 外部实时工具数据

### 8.2 最重要的质量要求
当系统证据不足时：
- 必须明确提示不确定性
- 不应假装准确
- 不应把未经清洗的网页内容直接当结论
- 不应把实时天气、票价、路线耗时写入静态知识库后假装长期可信

### 8.3 最不能接受的错误
- 关键词误判导致主流程走错
- 实时问题被当成静态知识回答
- 工具失败却输出像成功一样的结果
- 用户明确要求修改后，最终 itinerary 没有体现修改
- 缺少来源却用确定语气输出高风险信息

---

## 9. 成功标准
### 9.1 产品成功标准
- 用户能在网页端顺畅完成“输入需求 → 补齐信息 → 生成行程 → 继续修改”的闭环
- 用户能看到关键知识或实时信息的来源
- 系统能明确区分静态知识与实时数据
- 用户在常见旅游问题上，不需要反复纠正系统对问题类型的理解
- itinerary 结果整体做到“少修改可用”

### 9.2 核心观察指标
- 首轮问题类型路由是否稳定
- 用户继续追问修改的轮次
- 混合 RAG 的来源可用率和命中质量
- 工具调用成功率与失败透明度
- Reflection 发现并纠正明显问题的比例

---

## 10. 一句话总结
这是一个面向自由行用户、聚焦北京/上海/杭州试点的 Web 旅游 Agent：它通过聊天收集需求，结合知识检索、实时工具、工作记忆和最小反思闭环，持续生成并修改更可靠、更可执行的 itinerary。
