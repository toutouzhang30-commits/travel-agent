# 旅游 AI 助手 MVP 实施计划（RAG + Tool-Calling Agent 版）

## 1. 文档目的
本计划用于在当前仓库已有的“真实 DeepSeek 对话 + SSE + 最小行程生成/修改”基础上，重新定义后续实现顺序、目标架构和阶段验收标准。

这份计划**替代旧的“纯对话逐步增强 + 最后 PDF 导出”路线**。从现在开始，MVP 的目标不再是一个主要依赖模型自由生成的聊天接口，而是一个：

1. 具备 **Web 聊天界面** 的后端驱动应用
2. 具备 **RAG 检索能力** 的旅游知识代理
3. 具备 **真实工具调用能力** 的实时规划代理
4. 能够通过 **SSE 流式展示用户可见的代理执行过程** 的旅游助手

---

## 2. 本次架构调整的核心范围变化

### 2.1 新增能力
本次调整后，MVP 新增以下核心能力：

- **RAG 架构**
  - 用于存储和检索非结构化旅游攻略、城市背景、区域玩法、出行建议等知识
- **Tool Calling**
  - 通过 Spring AI 的 Function Calling 机制接入真实服务接口
  - 首批目标工具：
    - Maps / 路线与距离
    - Weather / 实时天气
    - Pricing / 门票或价格参考
- **Web UI**
  - 建立基于 Spring MVC + Thymeleaf/HTML/JS 的网页聊天室
  - 使用 SSE 流式展示聊天回复、检索动作、工具调用状态和最终结果

### 2.2 移除能力
以下内容不再属于 MVP 交付范围：

- **PDF 导出**
- 与 PDF 相关的 HTML 转换流程
- 任何以 PDF 为里程碑的阶段规划

### 2.3 保留但需要重构的能力
以下能力仍然保留，但其实现方式必须从“单纯 Prompt 编排”升级为“Agent + RAG + Tools”模式：

- 聊天式需求收集
- 动态追问
- 行程生成
- 行程修改
- 来源展示
- 不确定性提示

---

## 3. 当前仓库真实状态（作为迁移起点）

### 3.1 已存在的能力
当前仓库已经具备以下旧版能力：

- Spring Boot 应用可启动
- 已通过 DeepSeek OpenAI 兼容接口接入真实大模型
- 已提供同步聊天接口：`POST /api/chat`
- 已提供流式聊天接口：`POST /api/chat/stream`
- 流式接口基于 Spring MVC + `SseEmitter`
- 已存在最小旅游需求抽取与完整度判断
- 已存在最小结构化行程生成能力
- 已存在最小行程修改闭环
- 当前阶段仍通过排除数据源/JPA 自动配置来绕开数据库启动依赖

### 3.2 当前旧实现的关键限制
当前实现仍然属于“纯对话骨架”，与新目标架构之间存在明显差距：

- **没有 Web 页面**，只有 JSON/SSE API
- **没有 RAG 检索层**
- **没有向量存储**
- **没有真实工具调用**
- **没有来源引用结构闭环**
- **没有 Agent 事件流协议**
- **当前状态管理依赖前端把 `currentRequirement` / `currentItinerary` 原样回传**，不适合作为正式网页聊天室协议
- **`DeepSeekChatService` 同时承担模型调用、意图判断、业务编排**，职责过重，不利于接入 RAG 和 Tools

### 3.3 当前代码中与 PDF 相关的实际状态
经当前仓库代码与依赖检查：

- `src/main/java` 中**没有**真正落地的 PDF Controller / Service / Export 模块
- `pom.xml` 中**没有** PDF 导出依赖
- 当前需要移除的 PDF 内容主要仍停留在**文档层规划占位**，不是代码层实现

这意味着：

- 本次重构计划与架构文档时，直接移除 PDF 路线即可
- 不需要进行实际的 Java 代码删除
- 但后续仍需同步清理其他文档中残留的 PDF 目标描述

## 4. 新目标架构的核心决策

### 4.1 Web 层
- 统一使用 **Spring MVC**
- 继续使用 **SSE** 提供流式输出
- 建立 **Thymeleaf + HTML + JavaScript** 的网页聊天室
- 不引入 WebSocket
- 不同时维护 MVC 与 WebFlux 两套 Web 栈

### 4.2 AI 编排层
- 从“单个聊天服务”升级为 **Agent Orchestrator**
- 模型本身只负责：
  - 理解用户输入
  - 结合检索与工具结果组织回答
  - 输出结构化行程与说明
- 业务流程控制不再堆在单一 `DeepSeekChatService` 中

### 4.3 知识层拆分
后续知识来源分成三层，而不是都交给模型自由生成：

1. **结构化种子目录**
   - 城市、景点、标签、来源、可信度等稳定结构化信息
2. **RAG 知识库**
   - 非结构化攻略、城市背景、玩法建议、区域介绍
3. **实时工具数据**
   - 天气、价格、路线、距离、耗时等强时效信息

### 4.4 存储层
- **MySQL**：业务数据、会话、行程版本、来源引用
- **向量存储**：用于 RAG 检索
  - 当前决策：直接采用 **Spring AI PgVector**，满足跨重启持久化、检索扩展与后续演进需求
  - 通过 Spring AI 自动装配统一接入 embedding model 与 vector store
  - 向量维度、索引类型与 schema 初始化必须在配置层显式声明，避免运行时不一致

### 4.5 流式展示原则
网页端需要展示“代理在做什么”，但**不暴露模型不可控的内部隐藏推理**。

允许流式展示的内容应是用户可见、可验证的执行事件，例如：

- 正在补齐需求
- 正在检索杭州城市背景
- 正在调用天气工具
- 已拿到西湖景区天气信息
- 正在生成第 2 版行程
- 最终回复 / 行程结构 / 来源信息

不应把不可验证的完整内部 chain-of-thought 当作产品能力输出。

## 6. 分阶段执行说明

## Legacy 基线：当前旧实现（迁移起点）
### 现状说明
当前仓库的同步聊天、SSE、需求提取、最小 itinerary 和 itinerary modification 可以继续作为短期验证入口，但不再是最终架构。

### 迁移原则
- 保留已验证的 DeepSeek 接入与 SSE 技术路径
- 不再继续强化“前端回传整个状态对象”的协议
- 不再让 `DeepSeekChatService` 继续膨胀为总编排器

---

## Phase 1：基线收敛与接口重塑
### 目标
把当前“纯对话骨架”收敛为可承载 Web UI、RAG 和 Tool Calling 的基础设施。

### 给 AI 开发者的指令
1. 把 `application.yml` 中的模型 API key 改为环境变量读取。
2. 收敛到单一 Web 栈：
   - 移除 `spring-boot-starter-webflux`
   - 移除 `spring-boot-starter-webflux-test`
   - 保留 Spring MVC + SSE
3. 拆分当前 `DeepSeekChatService` 的职责：
   - 模型调用适配层
   - Agent 编排层
   - 会话/状态层
4. 重构聊天协议：
   - 从 `message + currentRequirement + currentItinerary`
   - 收敛为更适合网页聊天室的 `sessionId + message`
5. 定义新的流式事件协议，至少支持：
   - `status`
   - `retrieval`
   - `tool_call`
   - `tool_result`
   - `answer`
   - `itinerary`
   - `done`
   - `error`
6. 保留同步接口作为 debug / fallback 路径，但后续主路径以流式接口和网页聊天室为主。

### 本阶段输出物
- 环境变量化的模型配置
- 收敛后的 MVC Web 栈
- 新版聊天请求/会话协议
- Agent 事件流 DTO
- 轻量的会话上下文骨架（先内存态即可）

### 验证方式
- `./mvnw compile`
- `./mvnw test`
- `./mvnw spring-boot:run`
- `POST /api/chat`
- `POST /api/chat/stream`

### 不要做
- 不要在这一阶段就接入外部天气/地图/价格 API
- 不要在这一阶段就做向量入库
- 不要在这一阶段就做正式数据库持久化

---

## Phase 2：Web UI 与流式聊天室
### 目标
建立可直接演示的网页聊天界面，让用户能在浏览器里看到 AI 与 Agent 的流式执行过程。

### 给 AI 开发者的指令
1. 增加网页入口，例如：
   - `/`
   - `/chat`
2. 使用 Spring MVC 页面渲染返回 HTML。
3. 使用 Thymeleaf 或最小 HTML 模板生成聊天室页面。
4. 使用浏览器端 JavaScript 消费流式响应并展示：
   - 用户消息
   - AI 回复增量
   - 检索状态
   - 工具调用状态
   - 来源信息区
   - 最终 itinerary 卡片
5. 页面优先保证可用性，不做过度美化。
6. 保留 API 接口作为页面的后端能力，而不是把业务逻辑放到前端。

### 本阶段输出物
- 可打开的聊天页面
- 最小消息列表 UI
- SSE 流式渲染逻辑
- 最小 itinerary 展示区

### 验证方式
- `./mvnw spring-boot:run`
- 浏览器打开聊天页
- 输入一条需求后，能看到流式事件逐步展示

### 不要做
- 不要先做复杂前端框架
- 不要先做会话列表和账户体系
- 不要先做页面精装修

---

## Phase 3A：RAG 知识层最小闭环
### 目标
为 Agent 提供真实的知识检索能力，用于减少对城市背景、区域玩法、攻略摘要等内容的幻觉。

### 本阶段执行原则（当前最新决策）
- 本阶段优先完成 **“文档入库 -> 向量检索 -> 聊天链路消费检索结果”** 的最小闭环。
- 暂不要求一开始就提供完善的 RAG 方案。
- 暂不优先做现有 RAG 文档的重构、内容治理、精细切分或召回优化。
- 文档结构改善、chunk 策略优化、内容补强与召回质量调优，放到首版流程跑通并暴露真实问题后再做。

### 给 AI 开发者的指令
1. 建立最小知识文档模型，至少包含：
   - `city`
   - `topic`
   - `content`
   - `sourceName`
   - `sourceUrl`
   - `verifiedAt`
   - `confidenceLevel`
2. 建立最小可用的文档切分与入库流程，优先保证可入库、可检索，不先追求最佳切分效果。
3. 使用 **Spring AI PgVector** 完成首版可持久化检索闭环，并确保 schema、索引和维度配置与 embedding 模型一致。
4. 首版直接使用现有北京 / 上海 / 杭州文档作为知识源，不把文档重构作为前置条件。
5. 检索时至少支持：
   - 按城市过滤
   - 按主题过滤
   - 返回来源信息
6. 检索结果必须可进入最终响应与来源展示结构。
7. 首版完成后，再根据真实错误补做：
   - 文档结构优化
   - chunk 策略优化
   - 内容补强
   - 检索质量调优

### 本阶段输出物
- 向量存储配置
- 文档入库流程
- 最小检索服务
- 北京 / 上海 / 杭州的初始知识库

### 验证方式
- 能对“杭州适合轻松 3 日游吗”“上海拍照打卡适合住哪里”这类问题给出带来源的知识检索支撑
- 对应响应能够体现来源和校验时间

### 不要做
- 不要把所有内容都塞进向量库
- 不要把实时天气/价格这类信息伪装成 RAG 文档
- 不要先做复杂重排系统

---

### 本阶段完成说明
当前仓库已完成本阶段最小闭环，具体包括：
- Spring AI PgVector 向量存储配置已接通
- 北京 / 上海 / 杭州 RAG 文档已完成可入库结构整理
- 文档可自动入库并支持按城市过滤检索
- 聊天链路已能消费检索结果并返回来源信息
- 网页端已能展示检索状态与来源区

### 本阶段结论
Phase 3A 当前按“最小可用闭环”标准视为已完成。
但这只代表 RAG 能跑通，不代表 RAG 架构已经达到后续 Agent 可用标准。

## Phase 3B：高级 RAG、网页采集、多路召回与重排
### 目标
把 RAG 从“本地文档向量检索”升级为可治理、可扩展、可解释的知识系统，为后续 itinerary 生成、修改和 Reflection 提供更可靠的证据。

### 本阶段执行原则
- 网页采集是受控数据工程流程，不是运行时随意爬网页。
- RAG 必须支持多路召回，不能只依赖单路向量相似度。
- 检索结果进入回答或 itinerary 前必须经过重排和来源过滤。
- 来源、更新时间、可信度、内容 hash 必须进入元数据。
- 不把实时天气、实时票价、路线耗时伪装成 RAG 内容。

### 给 AI 开发者的指令
1. 新增受控网页采集层：
   - 使用来源白名单
   - 遵守 robots/站点规则和速率限制
   - 记录 `sourceUrl`、`sourceName`、`fetchedAt`、`verifiedAt`、`contentHash`
   - 不采集登录后、付费墙后、用户私有或权限不明的内容
2. 新增网页内容治理流程：
   - HTML 清洗
   - 正文抽取
   - 去重
   - 内容分段
   - 元数据标注
   - 低质量内容过滤
3. 建立高级 RAG 检索链路：
   - 向量召回
   - 关键词召回
   - 元数据过滤
   - 会话上下文召回
   - 重排
   - 上下文压缩
4. 重排信号至少包含：
   - 与用户问题的语义相关度
   - 与当前目的地和主题的一致性
   - 来源可信度
   - 更新时间
   - 内容片段完整性
5. RAG 响应必须继续返回可展示来源，而不是只返回拼接后的上下文文本。
6. `test.http` 需要补齐高级 RAG 手工验证用例。

### 本阶段输出物
- 受控网页采集模块设计
- 网页内容清洗与入库流程
- 多路召回服务
- 重排服务
- 上下文压缩策略
- 带来源的高级 RAG 验证用例

### 验证方式
- “杭州雨天怎么玩？”能召回雨天玩法、室内替代和来源信息
- “上海拍照打卡适合住哪里？”能结合区域、住宿和拍照主题召回
- “北京第一次去怎么安排？”能召回首次游玩节奏和经典区域建议
- 同一问题能看到多路召回后的重排结果，而不是只看单个向量命中
- 过旧或低可信来源不会优先进入最终上下文

### 不要做
- 不要运行时临时爬网页来回答用户
- 不要把未经清洗的网页正文直接塞进 Prompt
- 不要采集权限不明或登录后内容
- 不要把实时数据写进 RAG 知识库

---

### Phase 4 当前准备说明
当前进入 Phase 4 前，已具备以下前置条件：
- 页面已可展示 `tool_call` / `tool_result` 之外的完整事件链路
- RAG 已能处理静态知识问题，实时问题已能通过兜底分支与 RAG 边界区分
- 来源区和检索区已可复用到工具结果展示
- Phase 3B 的高级 RAG 可与 Phase 4A 并行推进，但实时工具不得依赖 RAG 伪造结果

Phase 4 的第一落点调整为 **Phase 4A：意图路由 + 受控 `@Tool` WeatherTool 闭环**。

新的主线原则是：

```text
模型做路由判断，代码做流程控制；
RAG 由编排器触发，实时数据由 @Tool 触发；
所有执行步骤都显式转成 SSE 事件。
```

关键词/限制词判断不再作为主分流方案。它只能作为极小的兜底保护，不应用来决定主要业务流程。

## Phase 4A：意图路由 + 受控 `@Tool` WeatherTool 闭环
### 目标
让 Agent 通过模型结构化路由替代关键词/限制词分流，并用受控 `@Tool` 调用打通 WeatherTool 的真实实时数据闭环。

### 本阶段执行原则
- 模型负责理解语义并输出结构化路由决策。
- 代码负责执行流程，不把业务流程完全交给模型自由发挥。
- 路由模型输出必须绑定结构化输出 schema，不允许用自由文本再做字符串解析。
- 结构化输出解析失败时必须走明确 fallback，不允许默默按普通对话继续。
- RAG 由编排器根据路由结果触发，用于静态/半静态旅游知识。
- 实时数据通过 Spring AI `@Tool` 暴露，例如天气、票价、路线。
- 工具调用采用受控执行，必须能显式生成 `tool_call` 和 `tool_result` 事件。
- 不再继续维护越来越长的关键词/限制词列表作为主判断方式。

### 阶段任务表

| 任务 | 目标 | 完成标准 |
| --- | --- | --- |
| 意图路由 | 新增 `IntentRoutingService`，让模型输出结构化路由决策 | 能区分需求补齐、RAG 问答、天气工具、行程生成、行程修改、暂不支持实时问题 |
| 结构化输出约束 | 路由必须使用 Spring AI structured output / BeanOutputConverter 等强约束 | 解析失败可检测并进入 fallback |
| RAG 触发重构 | 由编排器根据路由结果触发 RAG | “杭州雨天怎么玩”走 RAG；“杭州明天会下雨吗”不走 RAG |
| WeatherTool `@Tool` | 用 Spring AI `@Tool` 描述天气工具边界 | WeatherTool 有清晰输入/输出 DTO 与工具描述 |
| 受控工具执行 | 由代码捕获工具调用并执行工具 | 页面能看到 `tool_call` / `tool_result` |
| SSE 事件集成 | 保留用户可见执行过程 | 检索、工具调用、工具结果、最终回答均有结构化事件 |
| 测试用例 | 补齐边界验证 | `test.http` 覆盖 RAG、WeatherTool、未支持实时问题、行程生成和修改 |

### 给 AI 开发者的指令
1. 新增意图路由层，而不是继续扩展关键词/限制词分流：
   - 建议新增 `IntentRoutingService`
   - 建议新增 `AgentRouteDecision`
   - 建议新增 `AgentAction`
2. 路由必须绑定结构化输出：
   - 使用 Spring AI structured output / `BeanOutputConverter` / 等价强类型转换机制
   - 不允许让模型输出自然语言后再用字符串包含判断
   - 输出字段必须有默认值或显式 nullable 策略
   - 解析失败必须返回 `ROUTING_FAILED` 或进入明确 fallback
3. `AgentAction` 至少支持：
   - `COLLECT_REQUIREMENT`
   - `KNOWLEDGE_ANSWER`
   - `GENERATE_ITINERARY`
   - `MODIFY_ITINERARY`
   - `CALL_WEATHER_TOOL`
   - `UNSUPPORTED_REALTIME`
   - `ROUTING_FAILED`
4. 路由结果至少包含：
   - `action`
   - `requiresRetrieval`
   - `toolName`
   - `city`
   - `topic`
   - `reason`
   - `confidence`
5. 改造 RAG 触发：
   - 静态旅游知识、攻略、住宿区域、玩法建议走 RAG
   - 实时天气、实时票价、路线耗时不走 RAG
   - 行程生成或修改时，由编排器根据目的地和任务决定是否补充检索
6. 使用 Spring AI `@Tool` 定义 `WeatherTool`，而不是只做普通 Spring Bean 手动调用。
7. 工具调用采用受控执行：
   - 模型可请求工具调用
   - 应用代码负责执行工具
   - 执行前后显式发送 `tool_call` / `tool_result` SSE 事件
8. 工具结果要带最小来源与时间信息，例如：
   - 来源系统名
   - 请求时间
   - 生效时间/更新时间
9. 工具失败时要返回明确状态，不允许模型把失败伪装成成功结果。
10. 流式事件中要能显示：
   - 正在调用哪个工具
   - 工具成功/失败
   - 工具结果摘要
11. `PricingTool` / `MapsTool` 当前可以先返回明确“不支持实时查询”的不确定性提示，不要让 RAG 或模型编造。

### 本阶段输出物
- 意图路由服务与结构化路由 DTO
- RAG 触发主流程重构方案
- `WeatherTool` 的 `@Tool` 定义
- 受控工具执行服务
- `tool_call` / `tool_result` 事件闭环
- `test.http` 手工验证用例

### 验证方式
- “杭州雨天怎么玩？” -> RAG 检索，返回 `retrieval` + `answer`
- “杭州明天会下雨吗？” -> WeatherTool，返回 `tool_call` + `tool_result` + `answer`
- “上海拍照打卡适合住哪里？” -> RAG 检索，不调用工具
- “杭州 3 天，预算 3000，轻松一点” -> 生成 itinerary，并可附带 RAG 来源
- “第二天太累了，轻松一点” -> 修改当前 itinerary
- “上海迪士尼门票多少钱？” -> 返回 PricingTool 尚未接入的明确提示，不走 RAG 编造

### 不要做
- 不要使用 fake tool result 冒充真实外部数据
- 不要把工具调用细节硬编码在 Controller 中
- 不要让工具层和 Prompt 拼接层强耦合
- 不要继续把关键词/限制词列表作为 RAG 或 Tool 的主分流机制

## Phase 5：Agent 编排、Reflection 与行程生成重构
### 目标
把当前“需求抽取 -> 完整度判断 -> 生成 itinerary”的旧链路，升级为“会检索、会调用工具、能修改已有结果、能反思并纠错”的代理编排流程。

### 给 AI 开发者的指令
1. 引入统一的 Agent 编排入口，例如 `AgentOrchestratorService`。
2. 编排器至少能处理以下阶段：
   - 识别用户当前意图
   - 判断信息是否足够
   - 决定是否需要检索
   - 决定是否需要工具调用
   - 生成 itinerary
   - 修改 itinerary
   - Reflection 校验
   - 必要时局部修正
3. 保留当前已有的需求抽取、完整度判断、itinerary 生成与修改能力，但要让这些能力变成编排器下游能力，而不是总入口。
4. 生成和修改都必须能带回：
   - 当前需求状态
   - itinerary 版本结果
   - 来源引用
   - 不确定性说明
5. 对“第二天太累了，轻松一点”“不要博物馆，多安排吃的”“如果下雨怎么办”这类指令，优先在已有结果上调整，而不是完全重生成。
6. 增加 Reflection 闭环：
   - 检查结果是否满足用户核心约束
   - 检查天数、预算、节奏是否一致
   - 检查来源是否缺失、过旧或与结论不匹配
   - 检查工具失败是否明确提示不确定性
   - 检查用户修改请求是否真正反映到 itinerary
7. Reflection 只输出可验证的校验结论与必要修正，不暴露内部隐藏推理。

### 本阶段输出物
- Agent 编排服务
- 多轮会话上下文能力
- 与 RAG / Tool Calling 集成后的 itinerary 生成与修改逻辑
- Reflection 校验器
- 局部修正策略

### 验证方式
示例场景：
1. “杭州 3 天，预算 3000，轻松一点” → 生成 itinerary
2. “第二天太累了，轻松一点” → 修改现有 itinerary
3. “如果下雨怎么办” → 调用天气工具并给出替代建议
4. “上海周末拍照打卡适合住哪” → 检索城市知识并生成建议
5. “预算 1000，杭州 3 天，轻松一点” → Reflection 提醒预算/节奏风险或修正安排
6. 工具失败时 → Reflection 确认最终回答没有假装拿到实时数据

### 不要做
- 不要在这个阶段引入复杂 workflow 引擎
- 不要为了通用 Agent 平台化而过度抽象
- 不要让单个 Service 再次膨胀成不可维护的大类
- 不要把 Reflection 写成第二次完整重生成，优先做局部检查和局部修正
